# Week 9 — Hybrid retrieval + Ollama Q&A (RAG part 2)

**Status:** ✅ green

## Goal

Hybrid retrieval (sqlite-vec cosine + BM25 keyword, fused via RRF). `POST /memory/ask` streams an Ollama answer over SSE with citations.

## What landed

- `backend/src/meetwit/retrieval/hybrid.py`
  - `HybridRetriever` builds a BM25 in-memory index lazily from all `doc_chunks` (cached, invalidated on indexing events)
  - Issues a vector query via `embedding MATCH` on `doc_chunks_vec` AND a BM25 keyword query in parallel
  - **Reciprocal Rank Fusion** (RRF, k=60) combines the two ranks
  - Hydrates top-k chunks with their `Document` for citation labels
  - BGE query prefix applied: `"Represent this sentence for searching relevant passages: "`
- `backend/src/meetwit/llm/`
  - `client.py` — `ChatProvider` Protocol + `OllamaProvider` (httpx streaming, not the SDK)
  - `prompts.py` — `MEMORY_CHAT_SYSTEM`, `LIVE_ASSISTANT_SYSTEM`, `format_sources()`, `memory_chat_user_prompt()`
- `backend/src/meetwit/routers/memory.py`
  - `POST /memory/ask` → SSE stream with three event types:
    - `sources` (one event, full citation list as JSON)
    - `token` (one per generated token)
    - `done` (terminator)
  - `GET /memory/llm/status` → `{ollama_available, models}` for the Settings UI
- `main.py` lifespan instantiates `OllamaProvider(settings.ollama_url)` once
- Tests:
  - `test_retrieval.py` — module-scoped fixture indexes 3 small files; queries verify top-1 relevance + descending scores + empty-query handling

```bash

ollama pull qwen2.5:7b-instruct

# 2. Start the sidecar
cd backend && uv run python -m meetwit &

# 3. Index some docs
curl -X POST http://127.0.0.1:5167/knowledge/index-folder \
  -H 'content-type: application/json' \
  -d '{"folder":"/tmp/meetwit-docs"}'

# 4. Ask a question
curl -N -X POST http://127.0.0.1:5167/memory/ask \
  -H 'content-type: application/json' \
  -d '{"question":"What is our refund window?", "model":"qwen2.5:7b-instruct"}'
# → SSE stream: sources event first, then token stream, then done
```

## Open follow-ups

- BM25 corpus is rebuilt from scratch on first query. For 10k chunks this is ~200ms; for 100k it'd be ~2s. Move to FTS5 in V1.1.
- BM25 cache is in-memory only; survives sidecar restart by rebuilding. Fine for V1.
- No reranking layer (cross-encoder). RRF is good enough on small corpora; add reranker in V1.1 when chunk count > 50k.
