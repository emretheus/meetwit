# Week 3 — DB schema + Alembic + sqlite-vec

**Status:** ✅ green

## Goal

Full V1 SQLite schema, Alembic baseline migration that runs cleanly on a fresh DB, sqlite-vec extension loaded on every connection, vector roundtrip test confirms KNN works.

## What landed

- `backend/src/meetwit/models.py` — SQLAlchemy ORM for all relational tables:
  - `meetings` / `transcripts`
  - `documents` / `doc_chunks`
  - `summaries` / `decisions` / `action_items` / `conflicts`
  - `memory_chats` / `memory_messages`
  - `settings`
- `backend/src/meetwit/sqlite_vec_loader.py` — `vec0_loadable_path()` helper with `sys._MEIPASS` fallback for PyInstaller (risk #3 mitigation)
- `backend/src/meetwit/db.py` — engine factory registers `connect` event that loads sqlite-vec into every new connection
- `backend/alembic.ini` + `backend/src/meetwit/migrations/env.py` + `script.py.mako`
- `backend/src/meetwit/migrations/versions/2026_05_20_0001-v1_baseline_schema.py` — creates all 12 relational tables **and** the `doc_chunks_vec` virtual table (`USING vec0(chunk_id INTEGER PRIMARY KEY, embedding float[384])`)
- `backend/tests/test_schema.py` — runs alembic against a fresh tmp_path DB, verifies all 12 tables, inserts 100 random 384-dim vectors, runs KNN, asserts ordered results + <500 ms latency

## Bug fixed mid-week

The first version of `sqlite_vec_loader.py` validated the returned path with `Path(p).exists()` — but `sqlite_vec.loadable_path()` returns the **base** path without `.dylib`, and SQLite's `load_extension` auto-appends the platform suffix. The check now validates the parent directory exists instead.
