# Week 5 — ScreenCaptureKit Swift FFI

**Status:** ✅ green (compiles + symbols linked; runtime needs Screen Recording permission to fully test)

## Goal

System audio capture via Apple's ScreenCaptureKit, bridged from Swift to Rust via a C ABI. No virtual audio device (BlackHole) required.

## What landed

- `desktop/src-tauri/swift/SystemAudioTap.swift`
  - `@MainActor` `SystemAudioTap` class wrapping `SCStream`
  - Three `@_cdecl` exports: `meetwit_sck_available`, `meetwit_sck_start`, `meetwit_sck_stop`
  - Audio output handler converts `CMSampleBuffer` → interleaved f32 via `CMSampleBufferGetAudioBufferListWithRetainedBlockBuffer`
  - `excludesCurrentProcessAudio = true` so we don't capture Meetwit's own sounds
- `desktop/src-tauri/build.rs` — invokes `swiftc -emit-library -static -parse-as-library -O -target arm64-apple-macosx13.0` and `cargo:rustc-link-lib=`s the frameworks (ScreenCaptureKit, AVFoundation, CoreMedia, CoreAudio, Foundation) + Swift runtime (`swiftCore`, `swiftFoundation`)
- `desktop/src-tauri/src/audio/system.rs`
  - `extern "C"` block bindings to the three FFI fns
  - `SystemCapture` handle; `Arc::into_raw(SystemShared)` passed as `user_data`
  - Resampler shared across callbacks (rebuilt on rate change), buffered for fixed-chunk Rubato
  - `#![allow(unsafe_code)]` at the module level with a documented safety contract
- IPC commands: `system_audio_available`, `system_audio_start`, `system_audio_stop`, `system_audio_status`
- Frontend `SystemAudioControls` component with live RMS bar (purple to distinguish from mic green)

## What is NOT yet verified

- **Live capture end-to-end with permission granted.** Without an interactive macOS Screen Recording grant, SCK fails on first call. Manual test plan:
  1. `pnpm tauri:dev` → window opens
  2. Click "Start system audio"
  3. macOS prompts for Screen Recording. Approve via System Settings → Privacy & Security → Screen Recording → toggle Meetwit on
  4. **Fully quit and relaunch** the app (TCC only re-reads grants on relaunch)
  5. Click Start again → purple RMS bar should move when a YouTube video plays
- **Sample-alignment between mic + system audio.** Validated in Week 6 when the mix pipeline lands.
- **Notarization concerns** — the embedded Swift static library must be signed as part of the .app bundle. Validated in Week 15.

## Open follow-ups

- Replace the busy-wait `DispatchSemaphore` in `meetwit_sck_start`/`stop` with a proper async bridge when we have more Rust↔Swift surface. For V1 this is acceptable — the calls happen once at meeting start/stop.
- The `Arc` leaked into the C callback is not reclaimed on `Drop` (we don't keep the `*mut c_void`). The Swift singleton's `stop()` nils its callback ref so no further calls happen; the last `Arc` in `self.shared` drops naturally. Documented in the impl.
